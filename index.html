Git 是什么？

Git 是一款常用的版本管理软件，在企业生产中用得极其广泛。对于普通用户，用得最多的当然就是 GitHub 或者 Gitee 仓库的管理啦！

安装

前往 Git 官方下载页面，选择您的操作系统，下载安装即可。

配置

安装完成后，打开 Git Bash，输入以下命令设置 GitHub 账户和邮箱：


git config --global user.name "你的GitHub用户名"
git config --global user.email "你的GitHub注册邮箱"

输入下列命令查看是否设置成功：


git config user.name
git config user.email

本地仓库与远程仓库

  • 本地仓库：存放在电脑本地的文件夹；
  • 远程仓库：存放在 GitHub/Gitee 上的仓库。

利用 Git，可以顺利地将本地仓库推送到远程仓库，或从远程仓库拉取更新，非常便捷。

配置 SSH 秘钥


ssh-keygen -t rsa -C "你的GitHub注册邮箱"

一路回车即可，完成后在 C:\Users\你的用户名\.ssh 找到 id_rsa.pub 文件，将内容复制到 GitHub 的 SSH and GPG keys 页面，添加为新的 SSH key。

检测是否成功：


ssh git@github.com

若出现 successfully，说明绑定完成。

仓库克隆

在目标文件夹中打开 Git Bash，输入：


git clone git@github.com:用户名/仓库名.git

即可将远程仓库完整下载到本地。

仓库推送

  1. 在 GitHub 创建一个新仓库（可选择是否添加 README）。
  2. 在本地初始化并提交代码：

git init
git add .
git commit -m "首次提交"
git push

仓库拉取

当远程仓库被修改后，本地需要同步更新：


git pull

这样本地与远程就保持一致，避免推送冲突。

总结

通过本教程，你已经掌握了 Git 的最基本操作：安装、配置、SSH 秘钥、仓库克隆、推送与拉取。

未来你可能会需要更多分支管理和协作的技巧，那时可以继续学习进阶的 Git 知识。

为什么需要分支？

在团队开发中，每个人可能会同时开发不同的功能。如果大家直接在主分支 (main) 上提交，容易造成冲突和混乱。

分支 就像是复制一份主线代码，你可以在分支上自由修改，完成后再合并回主线。

常用分支命令


# 查看所有分支
git branch

# 创建新分支
git branch feature-login

# 切换到新分支
git checkout feature-login

# 创建并切换
git checkout -b feature-login

# 删除本地分支
git branch -d feature-login

推送与远程分支


# 推送本地分支到远程
git push -u origin feature-login

# 拉取远程分支
git checkout -b feature-login origin/feature-login

分支合并

完成开发后，将功能分支合并到主分支：


# 切回主分支
git checkout main

# 合并功能分支
git merge feature-login

合并后建议删除功能分支，保持仓库整洁。

解决冲突

当两个分支修改了同一文件的同一位置时，合并时会产生冲突，Git 会标记冲突区域：


<<<<<<< HEAD
这是主分支的内容
=======
这是功能分支的内容
>>>>>>> feature-login

手动修改为需要的内容后，再提交一次：


git add 文件名
git commit -m "解决合并冲突"

多人协作推荐流程

  1. 从主分支 main 拉取最新代码。
  2. 基于最新代码新建一个功能分支。
  3. 在功能分支上开发并提交。
  4. 推送功能分支到远程，并发起 Pull Request (PR)。
  5. 代码审查通过后，合并到主分支。

Rebase 与 Merge

Merge：直接把分支合并，保留分支历史，日志可能较复杂。

Rebase：把你的提交“平移”到主分支的最新提交之后，保持历史更整洁。


# 将功能分支基于 main 分支最新提交
git checkout feature-login
git rebase main

总结

通过学习分支和协作，你可以：

  • 独立开发新功能而不影响主分支
  • 用分支隔离不同任务
  • 通过合并或 rebase 保持代码历史清晰
  • 利用 PR 流程提升团队协作效率

这是成为 Git 高级用户 的必经之路 🚀

项目背景

假设我们要开发一个简单的网站项目，团队有三个人：小明、Alice 和 Bob。目标是使用 Git 和 GitHub 进行高效协作。

步骤一：初始化仓库


# 在本地创建文件夹并进入
mkdir my-website && cd my-website

# 初始化 Git 仓库
git init

# 创建一个 README 文件
echo "# My Website" > README.md

# 提交初始版本
git add README.md
git commit -m "初始化项目"

然后在 GitHub 创建一个远程仓库 my-website，把本地仓库推送上去：


git remote add origin https://github.com/username/my-website.git
git branch -M main
git push -u origin main

步骤二：成员克隆仓库

Alice 和 Bob 使用 git clone 把仓库拉到本地：


git clone https://github.com/username/my-website.git

步骤三：开发新功能

小明负责写首页，于是新建一个分支：


git checkout -b feature-homepage
echo "
欢迎来到我的网站
" > index.html
git add index.html
git commit -m "添加首页"
git push -u origin feature-homepage

在 GitHub 上就能看到 feature-homepage 分支。

步骤四：发起 Pull Request (PR)

  1. 小明进入 GitHub 仓库，点击 Compare & pull request。
  2. 填写描述，发起 PR 请求。
  3. Alice 和 Bob 进行代码审查，确认没有问题后点击 Merge。

步骤五：更新本地主分支

合并完成后，Alice 和 Bob 要同步最新代码：


git checkout main
git pull origin main

步骤六：解决冲突

如果 Alice 同时修改了 index.html，合并时可能出现冲突。解决方式：

  1. 打开冲突文件，手动修改冲突区域。
  2. 保存后执行 git add。
  3. 再执行 git commit 完成冲突解决。

最终成果

通过这个流程，团队实现了：

  • 使用分支隔离不同功能开发
  • 通过 PR 进行代码审查与合并
  • 保持主分支稳定
  • 及时同步和解决冲突

总结

这是一个最常见的 Git 协作流程：初始化 → 分支开发 → 提交 PR → 审查合并 → 更新主分支。熟悉后，你就能和团队高效协作啦 🚀。

常见问题 FAQ

Q1: 忘记提交代码就切换分支怎么办？

可以用 git stash 临时保存工作区：


git stash
git checkout new-branch
git stash pop

Q2: 不小心提交了错误的文件怎么办？

可以使用 git reset 撤销：


# 撤销暂存区的文件
git reset HEAD filename

# 回退到上一个提交
git reset --hard HEAD~1

Q3: Git 提交信息写错了怎么办？

如果只是最后一次提交，可以修改提交信息：


git commit --amend -m "新的提交信息"

注意：如果已经推送到远程仓库，请谨慎修改，以免影响其他人。

Q4: 如何回滚到某个历史版本？


# 查看提交记录
git log

# 回到指定提交
git checkout

Q5: GitHub 上分支太多怎么清理？

删除本地分支：


git branch -d branch-name

删除远程分支：


git push origin --delete branch-name

最佳实践

  • 使用分支：主分支保持稳定，新功能在 feature-xxx 分支开发。
  • 提交小而清晰：一次提交只做一件事，便于回滚和追踪。
  • 写有意义的提交信息：避免 "fix" 这种模糊的描述。
  • 定期同步远程分支：避免冲突堆积，保持本地最新。
  • 合并用 PR：不要直接推送到 main，通过 Pull Request 审查代码。
  • 使用 .gitignore：避免上传临时文件、日志、编译产物。
  • 定期清理分支：功能合并后删除分支，保持仓库整洁。

总结

Git 功能很强大，常见问题也容易解决。掌握以上 FAQ 和最佳实践，你就能更加顺畅地管理代码、与团队协作 🚀。

    • Git 进阶：使用 Rebase 打造整洁的提交历史

    发布日期：2025-08-22 ｜ 分类：Git ｜ 标签：#Git #Rebase #进阶

    在团队开发中，我们经常会遇到提交历史混乱的情况：分支合并一堆 merge commit， 历史线条交错，根本看不清楚每个功能的演进脉络。这时候，Git Rebase 就派上用场了。

    什么是 Rebase？

    简单来说，rebase 就是把一组提交“重新应用”到另一条分支上，从而得到一条更直的提交线。 和 merge 相比，它不会产生多余的合并提交，而是让历史看起来像是“线性开发”的。 

    
# 把当前分支的提交，基于 main 分支重新整理
git checkout feature
git rebase main

    Rebase 的好处

    • 让提交历史保持整洁，方便追溯问题。
    • 减少无意义的合并提交。
    • 更容易理解某个功能的完整开发过程。

    交互式 Rebase

    除了普通的 rebase，我们还可以用 git rebase -i 来交互式修改提交历史。 比如合并多个提交、修改提交信息、删除无效提交等。 

    
# 对最近 5 次提交做交互式 rebase
git rebase -i HEAD~5

    编辑器会弹出一个提交列表，你可以选择 pick、squash、 reword 等操作，把提交历史整理得干干净净。

    注意事项

    • ⚠️ 不要对公共分支使用 rebase，否则可能会让别人本地的历史混乱。
    • 在个人分支上用 rebase 整理提交是非常安全的。
    • 如果 rebase 出现冲突，解决后要 git rebase --continue。

    总之，Git Rebase 是一个强大的工具，能帮你打造清爽的提交历史。 学会它，你的 Git 技能就又进阶了一步 🚀。

    Git 标签（Tag）与版本发布

    发布日期：2025-08-22 ｜ 分类：Git ｜ 标签：#Git #版本管理 #Tag

    在软件开发中，我们常常需要为某个稳定版本打上标记，比如 v1.0、v2.0。 这时候就用到了 Git 标签（Tag）。标签就像是代码历史中的“快照”，能帮我们快速定位到某个版本。

    创建标签

    Git 有两种标签：轻量标签（lightweight）和附注标签（annotated）。

    
# 创建轻量标签 v1.0
git tag v1.0

# 创建附注标签（推荐）
git tag -a v1.0 -m "Release version 1.0"

    轻量标签只是对某个提交的引用，而附注标签则包含了作者、日期、描述信息等，适合发布版本。

    查看和管理标签

    
# 查看所有标签
git tag

# 查看特定标签的详情
git show v1.0

# 删除本地标签
git tag -d v1.0

    把标签推送到远程仓库

    
# 推送单个标签
git push origin v1.0

# 推送所有标签
git push origin --tags

    推送之后，团队成员就能在远程仓库（如 GitHub、GitLab）看到对应版本。 这也是开源项目发布版本的重要方式。

    签名标签

    如果你配置了 GPG，可以创建签名标签，保证版本的真实性和可信度。 

    
git tag -s v1.0 -m "Signed release 1.0"

    总结

    标签（Tag）是 Git 中非常实用的功能，它能帮助我们管理不同版本的发布节点。 在团队开发和开源项目中，打标签 + 发布说明 已经成为标准流程。

    如果你正在搭建一个项目，不妨试着为它打上第一个标签吧 🎉。

    Git 分支策略：Git Flow 与主干开发

    发布日期：2025-08-22 ｜ 分类：Git ｜ 标签：#Git #分支管理 #GitFlow

    当团队协作开发时，如何管理分支是一个关键问题。 不同的分支策略会直接影响开发效率和代码质量。今天我们来聊聊常见的两种分支模型： Git Flow 和 主干开发（Trunk Based Development）。

    Git Flow 工作流

    Git Flow 是一种经典的分支模型，适合大型项目和发布节奏较明确的团队。 它主要包括以下分支：

    • master：存放稳定可发布的版本
    • develop：主开发分支
    • feature/*：新功能开发
    • release/*：版本发布前的准备
    • hotfix/*：紧急修复
    
# 创建功能分支
git checkout -b feature/login

# 功能完成后合并回 develop
git checkout develop
git merge feature/login

    Git Flow 的优点是分工明确，缺点是分支较多，合并操作复杂。

    主干开发（Trunk Based Development）

    主干开发提倡所有开发者直接基于 主分支（通常是 main/master）进行开发， 并通过 短生命周期的分支 提交代码。每次合并都要尽快，避免长期分支。 

    
# 从 main 创建一个临时分支
git checkout -b fix-typo

# 修改完成后快速合并
git checkout main
git merge fix-typo

    主干开发强调持续集成（CI），每天多次合并，保证代码随时可发布。 适合敏捷团队和快速迭代的项目。

    对比与选择

    特点 Git Flow 主干开发
    适用场景 版本迭代周期长，团队较大 敏捷迭代，快速上线
    分支数量 多，结构清晰 少，开发集中
    学习成本 较高 较低

    总结

    分支策略没有绝对的对错，关键在于团队规模、项目需求和交付节奏。 如果你的团队较大，发布周期较长，可以选择 Git Flow； 如果你追求快速交付和敏捷迭代，主干开发 可能更适合。

    GitHub 协作流程：Fork、Pull Request 与 Code Review

    发布日期：2025-08-22 ｜ 分类：Git ｜ 标签：#GitHub #协作 #PullRequest #CodeReview

    在开源项目或团队协作中，GitHub 提供了一套完整的协作流程，让多人开发变得井然有序。 核心流程是 Fork → Pull Request → Code Review → Merge。

    步骤一：Fork 仓库

    如果你想为某个项目贡献代码，但没有写权限，可以先 Fork 仓库：

    1. 在目标仓库页面点击右上角 Fork，复制到你的 GitHub 账户下。
    2. 本地克隆你的 Fork 仓库：git clone https://github.com/你的用户名/仓库名.git

    步骤二：创建功能分支

    在本地仓库创建分支开发新功能或修复问题：

    
git checkout -b feature/fix-bug
# 修改代码
git add .
git commit -m "修复 bug"
git push origin feature/fix-bug

    步骤三：发起 Pull Request (PR)

    推送分支到远程后，在 GitHub 上发起 PR，请求将你的修改合并到主仓库：

    1. 进入你的 Fork 仓库，点击 Compare & pull request。
    2. 填写标题和描述，说明修改内容和原因。
    3. 选择目标分支（通常是原仓库的 main 或 develop 分支），提交 PR。

    步骤四：Code Review

    原仓库的维护者会对 PR 进行审查：

    • 检查代码风格、可读性和逻辑正确性
    • 提出改进建议
    • 如果通过，就合并到主分支；如果有问题，可以请求修改

    步骤五：合并与更新

    PR 被合并后，Fork 的本地仓库需要同步最新代码： 

    
# 添加原仓库为上游 remote
git remote add upstream https://github.com/原仓库/仓库名.git

# 拉取上游最新代码并合并
git fetch upstream
git checkout main
git merge upstream/main

# 推送到自己的远程 Fork
git push origin main

    最佳实践

    • 每个功能/修复用独立分支，避免直接在 main 上开发
    • PR 尽量小而精，便于审查
    • PR 描述清晰，说明修改原因与效果
    • 及时同步上游仓库，减少冲突
    • 参与 Code Review，提升团队协作水平

    总结

    熟练掌握 Fork + PR + Code Review 流程，你就能在 GitHub 上高效参与开源项目和团队开发。 这是现代开发者的必备技能，也是 Git 系列学习的重要补充 🚀。

    Git 故障排查与高效命令技巧

    发布日期：2025-08-22 ｜ 分类：Git ｜ 标签：#Git #技巧 #故障排查 #命令行

    Git 是一款强大的版本管理工具，但在日常使用中难免会遇到各种小问题。 本文总结一些常见故障排查方法和高效命令，让你在开发中更得心应手。

    1. 常见故障排查

    • 无法 push：remote rejected
      原因：远程仓库有新提交，本地未同步。
      解决： 
      
git pull --rebase origin main
git push
    • 冲突（Conflict）无法合并
      原因：本地修改与远程冲突。
      解决： 
      
# 查看冲突文件
git status

# 手动修改冲突
git add 冲突文件
git rebase --continue  # 或 git commit
    • 误删文件或提交
      解决： 
      
# 恢复未提交的文件
git checkout -- filename

# 回退到上一次提交
git reset --hard HEAD~1
    • 标签找不到
      解决： 
      
git fetch --tags
git tag

    2. 高效命令技巧

    • 查看提交历史： 
      
git log --oneline --graph --decorate --all
    • 快速切换分支： 
      
git checkout feature/login
# Git 2.23+ 可用：
git switch feature/login
    • 撤销命令快捷方式： 
      
git reset HEAD file      # 撤销暂存
git restore file          # 撤销修改
git restore --staged file # 撤销暂存
    • 快速删除本地和远程分支：

    1. Git 入门指南

    Git 是一款分布式版本控制系统，用于追踪文件修改、协作开发以及管理项目版本。与传统集中式版本控制相比，Git 的特点是速度快、分支管理灵活、历史记录完整。无论是代码开发，还是文档、笔记管理，都非常适用。

    1.1 基本概念

    • 本地仓库（Local Repository）：存放在你电脑本地的项目目录，保存所有文件的历史记录。
    • 远程仓库（Remote Repository）：存放在 GitHub、Gitee 或其他服务器上的仓库，可以同步本地仓库，实现团队协作。
    • 工作区（Working Directory）：你正在操作的文件夹，即你当前修改文件的位置。
    • 暂存区（Staging Area / Index）：用于存放即将提交到版本库的改动。

    1.2 Git 基本流程

    1. 在本地创建仓库或克隆远程仓库。
    2. 编辑文件，进行修改。
    3. 将修改添加到暂存区（git add）。
    4. 提交修改到本地仓库（git commit）。
    5. 将本地仓库的修改推送到远程仓库（git push）。
    6. 从远程仓库获取最新修改（git pull）。

    这个流程保证了本地和远程仓库的数据同步，支持多人协作。

    2. Git 安装与配置

    2.1 安装 Git

    在 Git 官方网站下载适合你操作系统的安装包：

    • Windows 下载
    • Mac 下载
    • Linux 下载

    安装过程中注意选择“使用 Git Bash”，方便命令行操作。

    2.2 配置用户名和邮箱

    安装完成后，打开 Git Bash，输入命令：

    
git config --global user.name "你的GitHub用户名"
git config --global user.email "你的GitHub注册邮箱"
git config user.name
git config user.email

    检查是否配置成功，如果返回正确的用户名和邮箱，说明配置完成。

    2.3 生成 SSH 密钥

    为了安全地连接 GitHub，需要生成 SSH 密钥：

    
ssh-keygen -t rsa -C "你的GitHub注册邮箱"

    一路回车即可，生成的公钥位于 C:\Users\你的用户名\.ssh\id_rsa.pub。

    复制公钥内容，打开 GitHub -> Settings -> SSH and GPG keys -> New SSH key，粘贴公钥完成添加。

    验证 SSH 连接：

    
ssh -T git@github.com

    出现 successfully authenticated 表示成功。

    4. Git 标签与版本发布

    Git 标签用于标记某个特定的提交点，通常用于发布版本或里程碑管理。标签分为两种：轻量标签和附注标签。

    • 轻量标签：只是在某个提交上打一个名字，没有额外信息。
    • 附注标签：带有作者、日期、说明信息的标签，类似一次完整的提交。

    创建标签示例：

    
# 创建轻量标签

    推送标签到远程仓库：

    
git push origin v1.0
# 或者一次性推送所有标签
git push origin --tags

    5. Git 分支策略与分支管理

    分支是 Git 的核心特性，用于并行开发、功能隔离和协作管理。

    5.1 常用分支策略

    • Git Flow：主干分支（master/main）用于生产，开发分支（develop）用于日常开发，功能分支（feature/*）独立开发后合并回 develop。
    • 主干开发（Trunk-Based Development）：所有开发尽量快速合并到主干，分支生命周期短，适合 CI/CD 快速迭代。

    5.2 分支操作

    
# 查看分支
git branch
# 创建新分支
git branch feature-login
# 切换分支
git checkout feature-login
# 合并分支
git checkout develop
git merge feature-login
# 删除分支
git branch -d feature-login

    建议在每次功能完成后及时合并并删除分支，保持仓库整洁。

    7.1 常见冲突解决

    当多个分支修改同一文件同一行时，会产生冲突，需要手动解决：

# 出现冲突后，手动编辑文件
git add 文件名
git commit -m "解决冲突"

    7.2 回滚操作

    • git reset --hard HEAD~1：回到上一个提交，丢弃改动。
    • git revert commit_id：生成一次反向提交，安全回滚。
    • git reflog：查看操作历史，找回误操作的提交。

    1. 环境准备与 Node.js 配置

    在开始搭建 Hexo 博客之前，需要确保本地环境具备 Node.js 和 Git，以下内容详细说明安装步骤及注意事项：

    1.1 Node.js 安装

    • 访问 Node.js 官方网站 https://nodejs.org/
    • 推荐安装 LTS（长期支持）版本，保证 Hexo 稳定运行。
    • 安装过程中选择“自动添加环境变量”，方便在命令行中调用 Node。

    安装完成后，可在终端检查版本：

    
node -v
npm -v

    Node.js 与 npm 是 Hexo 的核心依赖，缺一不可。

    1.2 Git 安装与配置

    访问 Git 官网 下载 Git，安装时选择默认选项即可。安装完成后，在终端输入：

    
git --version

    配置 Git 用户信息：

    
git config --global user.name "你的GitHub用户名"
git config --global user.email "你的GitHub注册邮箱"

    这是 Hexo 部署和提交 GitHub Pages 所必需的步骤。

    2. 安装 Hexo 并初始化博客

    安装 Hexo CLI：

    
npm install -g hexo-cli

    选择博客存放目录，执行：

    
hexo init my-blog
cd my-blog
npm install

    此时目录结构如下：

    • scaffolds/：文章模板
    • source/：博客内容
    • themes/：主题文件夹
    • _config.yml：博客配置文件

    可以在 _config.yml 中设置站点标题、描述、作者信息、语言、URL 等。

    3. 安装 Butterfly 主题

    进入主题目录，克隆 Butterfly：

    
cd themes
git clone https://github.com/jerryc127/hexo-theme-butterfly.git butterfly

    在博客主配置文件 _config.yml 中修改主题：

    
theme: butterfly

    3.1 Butterfly 主题功能简介

    • 响应式布局：支持移动端与桌面端自适应
    • 暗黑模式切换
    • 站点统计（访问量、阅读量）
    • 公告栏功能
    • 搜索、标签、分类过滤
    • 社交分享：微博、Twitter、Facebook 等
    • 评论系统：Valine、Gitalk、Twikoo 等

    4. 配置 Butterfly 主题

    打开 themes/butterfly/_config.yml，详细配置各项功能：

    4.1 基本信息

    
menu:
  home: /
  about: /about
title: 我的 Hexo 博客
subtitle: 学习与分享
author: 王学峰
language: zh-CN

    4.2 评论系统

    以 Valine 为例：

    
valine:
  enable: true
  appId: "你的LeanCloud AppID"
  appKey: "你的LeanCloud AppKey"
  placeholder: "说点什么吧..."

    Gitalk 配置示例：

    
gitalk:
  enable: false
  clientID: ""
  clientSecret: ""
  repo: ""
  owner: ""
  admin: []

    根据需要选择评论系统，开启后在文章下方显示评论区域。

    4.3 搜索功能

    
search:
  enable: true
  type: local

    开启搜索功能后，博客顶部会显示搜索框，支持文章标题、标签、内容搜索。

    4.4 站点统计

    
counter:
  enable: true
  type: busuanzi

    支持多种统计方式，如 busuanzi、Google Analytics 等。

    4.5 社交与分享

    
sns:
  weibo: "https://weibo.com/你的账号"
  twitter: "https://twitter.com/你的账号"
  github: "https://github.com/你的账号"

    配置后文章底部会显示社交图标，方便用户关注或分享文章。

    5. 本地预览与调试

    
hexo clean
hexo g
hexo s

    浏览器访问 http://localhost:4000，检查页面布局、功能是否正常。

    6. 部署到 GitHub Pages

    6.1 创建 GitHub 仓库

    登录 GitHub，新建仓库，例如 my-blog，选择 Public，添加 README。

    6.2 配置部署插件

    
npm install hexo-deployer-git --save

    修改博客主配置文件 _config.yml 部署段：

    
deploy:
  type: git
  repo: git@github.com:用户名/my-blog.git
  branch: main

    6.3 发布博客

    
hexo clean
hexo g
hexo d

    部署完成后，访问 https://用户名.github.io/my-blog/ 查看上线效果。

    7. 主题更新与备份

    • 更新主题：cd themes/butterfly && git pull
    • 备份博客：将 _config.yml、source、themes/butterfly/_config.yml 保存在本地或私有仓库
    • 多终端同步：使用 Git 管理博客目录，保证各电脑同步更新

    8. 常见问题与优化建议

    • 部署失败：检查 SSH 密钥，确保已添加到 GitHub
    • 图片不显示：使用绝对路径或 GitHub 仓库路径
    • 统计数据异常：确认外部统计服务开启
    • 评论系统不显示：检查 AppID、AppKey 是否正确
    • SEO 优化：修改 _config.yml 中描述、关键词、社交分享信息
